ROTP-PUF-FLASHER Web Flasher

Browser-based deployment tool for ESP32 IoT firmware

ESP32-WROOM-32D · MicroPython v1.25.0
Before you start: Use Google Chrome or Microsoft Edge and connect the ESP32 over USB. Steps 1 and 2 use DTR/RTS auto-reset. If the connection fails, hold BOOT while selecting the port and release it when the process starts. Steps 3 and later do not require BOOT. Important: Steps 3 to 6 reboot the ESP32 when they finish. If the device was already authenticated and sending data to the server, the previous Redis session can block re-authentication with HTTP 409. In that case, flush Redis before or after the operation; the serial monitor detects the condition and shows the exact command.

Chip Diagnostics

Detect the connected ESP32 model, revision, MAC address, and flash capacity. If the connection fails, hold BOOT while selecting the port.

2

Flash MicroPython

Flash MicroPython v1.25.0 to the ESP32 with esptool-js. The flow erases the full flash, writes the base image, reboots the board, and verifies that MicroPython enters Raw REPL before continuing. If the connection fails, hold BOOT while selecting the port and release it when the process starts.

3

Upload Files to the ESP32

Transfer the 18 IoT firmware .py modules to the ESP32 filesystem through Raw REPL. If a config.json is loaded, the tool also encrypts it with the runtime PUF key and uploads it automatically. The process usually takes 30 to 60 seconds plus PUF enrollment when needed. After a fresh flash, the first Raw REPL connection can take another 30 to 60 seconds; the tool waits automatically.

ROTP-PUF-FLASHER Firmware

Modular ESP32 firmware that reads temperature, humidity, and noise sensors; controls local actuators; and sends telemetry through puzzle-based authentication using HMAC-SHA256, AES-256-CBC, and JWT tokens.

Authentication Protocol
  1. Generate R2, 32 random bytes.
  2. Compute P2 = HMAC-SHA256(device_key || server_key, R2).
  3. Encrypt P2c = AES-256-CBC(P2, device_key, IV_random).
  4. Send puzzle_response to POST /api/v1/auth/device/login.
  5. Receive a JWT with 24-hour expiration and a single Redis session.
  6. Use the JWT as a Bearer token for POST /api/v1/device/reading.
Execution Flow
Boot main.py loads the encrypted configuration and validates required keys.
→
Initialize Device.py starts WiFi, NTP, sensors, actuators, and the local control loop.
→
Authenticate puzzle_auth.py builds the HMAC/AES puzzle response and obtains a JWT.
→
Telemetry http_client.py sends readings with a Bearer token while local actuator logic keeps running.
Known Limits
  • The noise sensor is used locally for LED control but is not sent to the API because the backend schema has no compatible field.
  • If the ESP32 reboots while the Redis session is still active, the device can be rejected with HTTP 409 until the session expires or is cleared.
  • config.json is encrypted at rest with a PUF-derived key, but without Secure Boot a physical attacker can replace the loader and extract runtime material.
  • Device-to-server communication uses HTTP in the lab path. The server can be placed behind TLS, but MicroPython urequests support is limited.
  • Authenticated OTA updates, hardware watchdog integration, and offline telemetry buffering are outside this release.
